## 🚨 Incident: Model sync problém (večer 19:30-20:00)

### Průběh:
1. **19:30** - Petr se ptal na lokální model pro Mac mini M4 16GB
2. **19:27** - Doporučil jsem qwen2.5:14b a začal stahování (`ollama pull qwen2.5:14b`)
3. **19:32** - Download dokončen (9.0 GB)
4. **Problém:** Model nebyl přidán do OpenClaw konfigurace
5. **Důsledek:** Gateway používal starý fallback stack → timeout/ticho
6. **19:59** - Petr si všiml, že jsem se odmlčel, ručně přepnul na Anthropic

### Root cause:
- Po `ollama pull` je potřeba model **explicitně přidat do `~/.openclaw/openclaw.json`**
- Stažený model byl dostupný v Ollama, ale OpenClaw o něm nevěděl
- Fallback chain stále používal `qwen2.5:7b` (starší verzi)

### Řešení (20:00-20:03):
1. ✅ Přidán `qwen2.5:14b` do `.models.providers.ollama.models[]`
2. ✅ Aktualizován fallback chain: `qwen2.5:7b` → `qwen2.5:14b`
3. ✅ Gateway restart přes `config.patch` API
4. ✅ Vytvořen helper skript: `sync-ollama-models.sh`

### Nová konfigurace:
```
Primary:    ollama/kimi-k2.5:cloud
Fallback 1: ollama/qwen2.5:14b         ← NEW!
Fallback 2: ollama/llama3.2-vision:11b
Fallback 3: anthropic/claude-opus-4-5
```

### Prevence:
- **Helper skript** vytvořen: `~/.openclaw/workspace/scripts/sync-ollama-models.sh`
- Automaticky synchronizuje všechny Ollama modely do OpenClaw config
- Použití po každém `ollama pull <model>`
- README: `scripts/README-sync-ollama.md`

### Lessons learned:
- Po stažení nového modelu VŽDY zkontrolovat/updatovat OpenClaw config
- Vytvořit helper nástroje pro opakující se administrativní úkoly
- Lepší error handling když model není v configu
